# Using orator

Orator generates Rust server stubs (axum) from an OpenAPI 3.1 spec. You edit the
spec, run orator to regenerate the typed scaffolding, then implement the
generated traits on your own struct. **You never hand-edit generated files.**

## Workflow

1. Edit the OpenAPI spec.
2. Regenerate: `orator <spec> --output <dir>` (see CLI below).
3. Implement the generated trait(s) on your own type.
4. To change the shape of the API, go back to step 1 — do not edit files under
   the output directory.

## CLI

```bash
orator openapi.yaml --output src/api/generated
```

- `<spec>` is a **positional** argument (YAML or JSON).
- `--output` / `-o` (required): directory for generated code.
- `--no-header-params`: disable header parameter extraction.
- `--no-cookie-params`: disable cookie parameter extraction.

There is **no** `--framework` flag and **no** `generate` subcommand. axum is the
only target.

## Implementing the generated code

Orator generates **one trait per OpenAPI tag** (e.g. `MembersApi`,
`BookingsApi`) in `<output>/operations.rs`. Implement each trait on your own
struct:

```rust
use crate::api::generated::operations::{GetMemberPath, GetMemberResponse, MembersApi};

struct MyApi;

impl MembersApi for MyApi {
    // Your error type. The axum layer requires it to implement IntoResponse.
    type Error = std::convert::Infallible;
    // Per-request context threaded in from the axum layer (auth, db handles, …).
    type RequestContext = ();

    async fn get_member(
        &self,
        _ctx: (),
        path: GetMemberPath,
    ) -> Result<GetMemberResponse, Self::Error> {
        match self.lookup(path.member_id) {
            Some(member) => Ok(GetMemberResponse::Ok(member.into())),
            None => Ok(GetMemberResponse::NotFound(/* the generated Error type */)),
        }
    }
}
```

Key rules:

- **Each method takes `&self`, then `ctx: Self::RequestContext`, then one struct
  per parameter group**: `XxxPath`, `XxxQuery`, `XxxHeader`, `XxxCookie`, and a
  body type for request bodies. Only the groups an operation actually uses are
  present.
- **Pick `type Error` and `type RequestContext`** once per trait impl. `()` is a
  fine `RequestContext` if you need nothing; `Infallible` is a fine `Error` if
  you never fail.
- **Every status code in the spec becomes a variant** of the operation's
  `XxxResponse` enum (`Ok`, `Created`, `NotFound`, `NoContent`, …), carrying that
  response's body type. Return the matching variant wrapped in `Ok(...)` —
  **even for error statuses** (`Ok(GetMemberResponse::NotFound(err))`).
- **A spec `default` response maps to a special `Default(StatusCode, BodyType)` variant** (e.g. `Default(orator_axum::http::StatusCode, Error)`) that carries the runtime HTTP status — use it for catch-all, status-bearing responses.
- **Reserve `Err(Self::Error)`** for unexpected failures you want the axum layer
  to turn into a response via `IntoResponse` — not for modelled error statuses.

## Don't

- Don't edit anything under the output directory — it carries a
  `// Generated by orator <version>` / `// Do not edit manually.` header and will be
  overwritten on the next run. Change the spec and regenerate instead.
